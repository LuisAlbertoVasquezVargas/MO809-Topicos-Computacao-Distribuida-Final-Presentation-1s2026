\section{Proposed Data Structures}
\begin{frame}{Data Structures: The Hash Tree}
    \begin{itemize}
        \item \textbf{Structure:} Internal nodes contain hash tables; leaves store candidate itemsets.
        \item \textbf{Traversal Logic:} 
        \begin{enumerate}
            \item Hash the $i$-th item of a transaction.
            \item Follow the corresponding branch.
            \item Repeat until a leaf is reached and verify the candidate.
        \end{enumerate}
        \vfill
        \item \textbf{The Spark Bottleneck:}
        \begin{itemize}
            \item \textbf{High Collision:} Multiple itemsets mapping to the same leaf force slow linear searches.
            \item \textbf{CPU Overhead:} Repeated hashing at every level is expensive when scaled across RDD partitions.
        \end{itemize}
    \end{itemize}
\end{frame}

\begin{frame}{Data Structures: The Trie (Prefix Tree)}
    \begin{itemize}
        \item \textbf{Structure:} A rooted tree where each node represents a single item.
        \item \textbf{Mechanism:} 
        \begin{itemize}
            \item Paths from root to leaf represent full itemsets.
            \item Common prefixes are shared, reducing redundant storage.
        \end{itemize}
        \vfill
        \item \textbf{Advantage in RDDs:}
        \begin{itemize}
            \item \textbf{Zero Hashing:} Navigates purely by pointers/references.
            \item \textbf{Predictable Traversal:} If a prefix isn't in the transaction, the entire branch is pruned instantly.
            \item \textit{Trade-off:} Searching for a child node among many siblings can become $O(k)$ where $k$ is the alphabet size.
        \end{itemize}
    \end{itemize}
\end{frame}

\begin{frame}{Data Structures: The Hash Table Trie}
    \begin{itemize}
        \item \textbf{The Hybrid Concept:} 
        \begin{itemize}
            \item A Trie where each node uses a \textbf{local Hash Table} to store its children.
        \end{itemize}
        \vfill
        \item \textbf{Why it dominates in Spark:}
        \begin{itemize}
            \item \textbf{Constant Time ($O(1)$):} Finding the next item in a sequence doesn't require scanning a list of siblings.
            \item \textbf{CPU Efficiency:} It combines the pruning power of a Trie with the lookup speed of Hashing.
            \item \textbf{Memory-Speed Tradeoff:} Uses more RAM than a standard Trie, but in an RDD environment, RAM is exactly what we have to spare.
        \end{itemize}
    \end{itemize}
    \vfill
    \begin{alertblock}{The Verdict}
        The Hash Table Trie fixes the ``branching bottleneck'' of standard Tries while maintaining better cache locality than traditional Hash Trees.
    \end{alertblock}
\end{frame}
